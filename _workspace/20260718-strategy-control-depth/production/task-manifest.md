## Acceptance definitions

- **Map scale:** the final-content constants are exactly 24×12, yielding 288 gross cells and 2.25× the current 16×8 area across all 10 live stages. A 40×24 map is a non-content stress fixture only.
- **Material route/zone:** exactly three routes per stage; each connects staging to pressure, has ≥50% nonshared cells between shared junctions, ≥1 distinct affordance, ≥4-cell intended combat frontage, two reconnects/stage, and remains independently connected if another route is blocked. Longest default route≤1.35× shortest; measured success share 15–70%; static camp≤60%.
- **Control responsiveness:** event timestamp to first visible or accessibility-announced accepted/rejected feedback. At least 500 samples per device-input-renderer profile; p50/p95/hard max≤50/100/200 ms; zero missing/duplicate actions; cumulative drag 6 CSS px mouse/pen or 12 CSS px touch and tap≤500 ms.
- **Renderer parity:** same normalized input sequence and checkpoint must yield identical navigation, accepted/rejected action order, semantic encounter events, final campaign state, trace, canonical save bytes, retry, suspension and fallback cutover in Three.js and forced Canvas2D. Pixel identity is not required.
- **Frame independence:** fixed 1/60 s step, `.10 s` delta clamp and six catch-up steps; same ten-second path at 30/60/120 Hz ends within `.02` world units. Camera follow uses `τ=.1304 s`, target delta≤`.001` world units and t90 spread≤35 ms.
- **Deterministic save:** all 10 stage checkpoints and all live reward branches; at least 1,000 seeded sequences×150 operations; restore/recreate state and canonical envelope bytes match; incompatible schema/rules/impossible traces accepted zero times.
